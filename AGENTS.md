# Agents guide

Коротка точка входу для AI-агентів і контриб'юторів, які працюють з цим репо.

## Project

Static Next.js (App Router) сайт для каталогу інженерних калькуляторів. Контент — JSON-driven ([`data/content.json`](data/content.json)), TypeScript, Vitest. Деталі стеку, локальний dev і деплой — у [README.md](README.md).

## Editing content

Майже весь видимий контент сайту (калькулятори, категорії, IVapps top bar, сторінка `/author`, бренд-копія) живе в одному файлі: [`data/content.json`](data/content.json).

Як додавати/редагувати — див. **[docs/content-editing.md](docs/content-editing.md)**. Гайд містить:

- схему запису калькулятора з описом усіх полів
- таблицю display modes (`embed` / `external` / `modal`) і коли який обрати
- як додати категорію (включно з оновленням `CategorySlug` union у [`lib/calculators.ts`](lib/calculators.ts))
- де редагувати IVapps top bar, утилітарні посилання, тексти `/author` і бренд-копію
- іконки (lucide-react)
- перевірочні команди й найчастіші помилки

## Planning calculation algorithms

Перед implementation plan для нового нормативного алгоритму потрібно окремо погодити з користувачем алгоритм звіту. Працюй малими кроками: один пункт звіту або один блок вихідних даних за раз. Не переходь до плану реалізації, поки користувач явно не погодив:

- перелік вихідних даних, їхні назви в UI, позначення, одиниці та умови показу;
- кожен підпис кроку звіту;
- нормативне посилання для кожного кроку;
- точний plain-text рядок формули: символічний запис, числова підстановка, результат і одиниці;
- правила для табличних значень, приміток норм, граничних умов, warning/error;
- порядок кроків звіту та кроки, які показуються тільки за певних умов.

Після погодження створи окремий файл-контракт у `docs/superpowers/specs/` з назвою на кшталт `YYYY-MM-DD-<calculator>-report-contract.md`. У ньому зафіксуй тільки погоджений текст звіту: captions, display conditions, items, formulas, warning/error text. Основна design spec має посилатися на цей contract-файл як на джерело істини. Implementation plan і тести повинні брати формули саме з contract-файлу; якщо під час реалізації потрібно змінити текст або формулу, спочатку онови contract і повторно погодь зміну.

Для пункту **вихідних даних** у звіті:

- показуй саме ті дані, які задає користувач або які важливі для аудиту прийнятої схеми;
- не показуй як "вихідні" коефіцієнти, що обчислюються пізнішими кроками;
- для умовних полів явно фіксуй правило показу, наприклад `IL` тільки для глинистих ґрунтів або глинистого заповнювача;
- для ручних режимів пояснюй, що значення прийняті користувачем, і не дублюй автоматичні поля, які не впливають на розрахунок у цьому режимі.

Для табличних значень та інтерполяцій:

- перед застосуванням коефіцієнта за таблицею покажи, які вхідні ознаки визначають рядок/графу таблиці: тип ґрунту, `IL`, `L/H`, конструктивна схема тощо;
- якщо значення береться напряму з таблиці, вкажи рядок/графу та прийняте число;
- якщо потрібна інтерполяція, окремо покажи обидва табличні значення, між якими інтерполюємо, і формулу інтерполяції з підстановкою;
- не приховуй примітки норм: якщо примітка змінює значення або застосовність формули, вона має бути окремим кроком, warning або явною фразою в кроці;
- не вводь проміжні позначення, яких немає в нормі, лише для зручності, якщо користувач це не погодив. Якщо велика формула лишається однією формулою в нормі, звіт має показувати її як одну формулу з повною підстановкою.

## Verification

Перед комітом:

```bash
npm run typecheck
npm run build
```

Для візуальної перевірки — `npm run dev` і http://localhost:3000.
