# Calculation Reporting Guide

Універсальна інструкція для додавання нативних калькуляторів із покроковим розрахунковим звітом. Оновлюй цей документ щоразу, коли з'являється новий повторюваний патерн, вимога до нормативних посилань, формат формул або перевірка.

## Принцип

Калькулятор має відокремлювати:

- розрахункове ядро в `lib/<calculator>.ts`
- тести розрахунку в `lib/<calculator>.test.ts`
- React UI в `components/calculators/<calculator>-calculator.tsx`
- реєстрацію в `lib/calculators.ts`, `components/calculator-shell.tsx`, `data/content.json`
- стилі в `app/globals.css`

UI не повинен містити інженерної логіки. Він тільки збирає вихідні дані, викликає розрахункове ядро й рендерить результат.

## Обов'язковий порядок роботи

1. Узгодити з користувачем усі вихідні дані.
2. Узгодити кожен підпис над формулою.
3. Узгодити нормативне посилання для кожного кроку.
4. Узгодити формат формульного рядка: символи, підстановка, результат, одиниці.
5. Записати специфікацію в `docs/superpowers/specs/`.
6. Записати implementation plan в `docs/superpowers/plans/`.
7. Додати failing tests.
8. Реалізувати розрахункове ядро.
9. Додати UI з математичним рендерингом формул.
10. Зареєструвати калькулятор у каталозі.
11. Запустити `npm run test`, `npm run typecheck`, `npm run build`.

## Розрахункове ядро

Кожен калькулятор зі звітом має експортувати:

```ts
export const CALCULATOR_NOTATION = {
  // централізований словник позначень
} as const;

export type CalculatorInput = {
  // нормалізовані вихідні дані
};

export type CalculatorReportStep = {
  key: string;
  caption: string;
  formula?: string;
  items?: string[];
};

export type CalculatorReport = {
  input: CalculatorInput;
  valid: boolean;
  errors: string[];
  warnings: string[];
  values: CalculatorValues | null;
  steps: CalculatorReportStep[];
};
```

Правила:

- усі формули рахувати в базових одиницях
- одиниці явно вказувати в `formula` і `items`
- `formula` зберігати як plain-text ланцюжок для тестів, accessibility і копіювання
- UI має рендерити цей самий зміст математично, а не як звичайний текст
- кожен нормативний пункт має бути в `caption`
- якщо формула є геометричним виведенням із пунктів норм, це прямо вказувати в підписі або специфікації

## Формат кроку звіту

Кожен розрахунковий крок має мати:

```text
<Підпис що обчислюється> (<пункт і документ>):
<symbol> = <symbolic formula> = <numeric substitution> = <result unit>
```

Приклад:

```text
Визначення другої складової мінімальної площі армування As,min,2 (п. 8.2.1.1 ДСТУ Б В.2.6-156:2010 / п. 9.2.1.1(1), формула 9.1N EN 1992-1-1):
As,min,2 = 0.0013 * bt * d = 0.0013 * 1000 * 450 = 585 мм² = 5.85 см²
```

Для табличних даних:

```text
Визначення середньої міцності бетону на осьовий розтяг fctm за класом бетону (табл. 3.1 ...):
fctm = 2.9 МПа (C30/37)
```

## Математичний стиль формул

У UI формули потрібно рендерити математично:

- `As,min` як `A` з індексом `s,min`
- `As,min,1` як `A` з індексом `s,min,1`
- `fctm` як `f` з індексом `ctm`
- `fyk` як `f` з індексом `yk`
- `bt` як `b` з індексом `t`
- `a_s` як `a` з індексом `s`
- `Øs` як `Ø` з індексом `s`

Використовуй `components/calculators/math-notation.tsx` або локальний renderer, який централізовано перетворює словник позначень у JSX. Не дублюй ручну розмітку одних і тих самих символів у багатьох місцях.

Plain-text формула залишається в DOM як `aria-label` або еквівалентний доступний текст, щоб:

- тест перевіряв точний погоджений рядок
- screen reader мав повний текст
- користувач міг скопіювати зміст без втрати символів

Формули у звіті не потрібно обводити рамками або заливками. Основний акцент має бути на підписі кроку та читабельному математичному рядку. Перевірки застосовності норм також використовують спокійний inline-стиль.

## Креслення та позначення

Якщо калькулятор використовує геометричні величини, додай схему-пояснення:

- окремо для кожного типу конструкції або розрахункової схеми
- з підписами тих самих позначень, що й у словнику
- у SVG або іншому контрольованому векторному форматі
- з `role="img"` і зрозумілим `aria-label`

Схема має показувати тільки потрібні для розрахунку величини. Не додавай зайві елементи, якщо вони не використовуються у формулах або вихідних даних.

## Handoff між калькуляторами

Якщо результат одного калькулятора є вихідними даними для іншого, додай пряме посилання з query parameters.

Приклад:

```text
/calculator/rebar-area-bars?minimumArea=678.6&unit=mm2
```

Правила:

- параметри мають бути стабільними й документованими в коді приймаючого калькулятора
- приймаючий калькулятор має безпечно валідувати query parameters
- якщо параметри невалідні, використовуй стандартні значення
- біля посилання потрібно показувати не тільки переданий результат, а й короткий результат приймаючого розрахунку, якщо його можна обчислити тією самою спільною логікою без відкривання іншого калькулятора
- додавай `returnTo` і `returnLabel`, якщо користувач природно має повернутися до попереднього розрахунку
- приймаючий калькулятор має показувати кнопку/посилання назад, якщо `returnTo` валідний
- посилання має бути видимим поруч із результатом, а не тільки внизу сторінки
- тести мають перевіряти і `href`, і фактичне заповнення приймаючого калькулятора

## Валідація

Кожен калькулятор має повертати стабільний звіт навіть при невалідних даних:

- `valid: false`
- список `errors`
- вихідні дані, які користувач ввів
- без `NaN` у формулах

Типові перевірки:

- додатні геометричні величини
- межі застосовності норм
- наявність класів матеріалів у довідниках
- сумісність вхідних параметрів

Якщо один нормативний блок не застосовується, але інший залишається чинним, не зупиняй увесь розрахунок. Додай `warning` і продовжуй ту частину, яка дозволена.

## Тести

Мінімальний набір тестів:

- окремі формули
- табличні значення з довідників
- порядок `steps`
- точні `caption` або ключові нормативні посилання
- точні plain-text `formula`
- граничні умови застосовності норм
- валідація помилкових даних
- UI smoke test через `CalculatorShell`

Тест має спочатку падати, потім проходити після реалізації.

## Реєстрація калькулятора

Для нового native-калькулятора:

1. Додай literal у `nativeCalculator` union в `lib/calculators.ts`.
2. Імпортуй компонент у `components/calculator-shell.tsx`.
3. Додай `case` у `NativeCalculator`.
4. Додай запис у `data/content.json` з `displayMode: "native"`.
5. Перевір `openUrl`, `order`, `seoTitle`, `seoDescription`, `useCases`, `tags`.
