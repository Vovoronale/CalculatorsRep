# Calculation Reporting Guide

Універсальна інструкція для додавання нативних калькуляторів із покроковим розрахунковим звітом. Оновлюй цей документ щоразу, коли з'являється новий повторюваний патерн, вимога до нормативних посилань, формат формул або перевірка.

## Принцип

Калькулятор має відокремлювати:

- розрахункове ядро в `lib/<calculator>.ts`
- тести розрахунку в `lib/<calculator>.test.ts`
- React UI в `components/calculators/<calculator>-calculator.tsx`
- реєстрацію в `lib/calculators.ts`, `components/calculator-shell.tsx`, `data/content.json`
- стилі в `app/globals.css`

UI не повинен містити інженерної логіки. Він тільки збирає вихідні дані, викликає розрахункове ядро й рендерить результат.

## Descriptions for input fields

Кожне поле схеми вводу повинно мати `description`, який відображається через кнопку `?`. Підказка має допомагати прийняти правильне інженерне рішення, а не лише повідомляти номер пункту норми.

Будуй опис у такому порядку:

1. **Що ввести або обрати.** Назви фізичний чи конструктивний зміст величини. Поясни, де її взяти: з розрахункової моделі, креслення, специфікації, сертифіката матеріалу або проєктної документації.
2. **Як значення використовується.** Назви показник, табличний рядок, коефіцієнт, перевірку або формулу, на які воно впливає. Якщо вибір змінює видимість інших полів, скажи про це.
3. **Нормативне джерело.** Наведи документ і максимально точну адресу: пункт, таблицю, позицію чи графу, примітку, рисунок або формулу.

Базовий шаблон:

```text
<Що задати і звідки взяти>. <Як це впливає на розрахунок>. Джерело: <документ, пункт/таблиця/позиція/примітка>.
```

Правила за типом поля:

- **number** — поясни, який саме результат розрахунку вводиться, для якого перерізу чи комбінації, допустимий знак, одиниці та граничні умови;
- **select/radio** — поясни зміст варіантів і наслідок вибору, а не переказуй їхні назви;
- **checkbox** — явно сформулюй умову, коли слід обрати «Так», і кількісний або логічний наслідок;
- **умовне поле** — поясни причину його появи й зафіксуй `display condition` у report contract;
- **ручний режим** — зазнач, яку автоматичну нормативну перевірку вимкнено, де використовується введене число і що обґрунтування значення залишається за користувачем;
- **табличне значення** — назви ознаки, що визначають рядок і графу; якщо користувач обирає рядок сам, попередь, що калькулятор не перевіряє його застосовність автоматично.

Не використовуй внутрішні enum-коди, назви змінних або технічні деталі реалізації. Не пиши підказки на кшталт `ДБН ..., таблиця 5.1` без пояснення призначення поля.

Погано:

```text
ДБН В.2.6-198:2014, таблиця А.2, показник S1.
```

Добре:

```text
Оберіть клас наслідків, встановлений у проєктній документації. Він формує показник S1 і впливає на групу конструкції. Джерело: таблиця А.2 ДБН В.2.6-198:2014.
```

Для нормативного калькулятора погоджені описи полів фіксуй у report contract разом із labels, options, defaults і display conditions. Тести повинні охоплювати всі гілки схеми та перевіряти, що кожне видиме поле має практичну інструкцію і точне нормативне посилання; одного smoke-тесту default-стану недостатньо.

## Обов'язковий порядок роботи

1. Узгодити з користувачем усі вихідні дані.
2. Узгодити кожен підпис над формулою.
3. Узгодити нормативне посилання для кожного кроку.
4. Узгодити формат формульного рядка: символи, підстановка, результат, одиниці.
5. Записати окремий report contract у `docs/superpowers/specs/YYYY-MM-DD-<calculator>-report-contract.md`.
6. Звірити report contract із погодженими повідомленнями й отримати підтвердження користувача, що погоджені пункти перенесені коректно.
7. Записати design spec у `docs/superpowers/specs/` з посиланням на report contract як на джерело істини.
8. Записати implementation plan в `docs/superpowers/plans/`.
9. Додати failing tests.
10. Реалізувати розрахункове ядро.
11. Додати UI з математичним рендерингом формул.
12. Зареєструвати калькулятор у каталозі.
13. Запустити `npm run test`, `npm run typecheck`, `npm run build`.

## Report Contract Gate

Для нового нормативного калькулятора report contract є результатом роботи з погодження по кроках. Не починай implementation plan, поки contract не створений і не підтверджений користувачем.

Report contract має містити тільки погоджені речі:

- UI labels, option labels, default values і правила показу полів;
- порядок кроків звіту;
- короткий `caption` для кожного кроку та окремий `normativeBasis` з нормативним посиланням;
- `display condition` для умовних кроків;
- `items` для вихідних даних і табличних пояснень;
- `notes` для застосовності норм, рядків/граф таблиць, приміток і пояснень;
- точні plain-text `formula` або `formulas`;
- warning/error text;
- handoff/query parameters між калькуляторами, якщо результат передається далі.

Шаблон верхнього блоку contract-файлу:

```text
# <Calculator Name> Report Contract

Date: YYYY-MM-DD
Calculator: `<calculator-slug>`
Status: Agreed source of truth for report text and formulas

This file contains the agreed UI labels, report captions, display conditions,
items, notes, formula strings, warning/error text, and handoff parameters.
Implementation plans, tests, and code must treat this file as canonical.
If any report wording or formula changes during implementation, update this
file first and get approval before changing tests or code.
```

Design spec може коротко описувати архітектуру й UX, але не замінює contract. Якщо design spec і report contract розходяться, правильним вважається report contract.

## Розрахункове ядро

Кожен калькулятор зі звітом має експортувати:

```ts
export const CALCULATOR_NOTATION = {
  // централізований словник позначень
} as const;

export type CalculatorInput = {
  // нормалізовані вихідні дані
};

export type CalculatorReportStep = {
  key: string;
  caption: string;
  normativeBasis?: string;
  formula?: string;
  items?: string[];
};

export type CalculatorReport = {
  input: CalculatorInput;
  valid: boolean;
  errors: string[];
  warnings: string[];
  values: CalculatorValues | null;
  steps: CalculatorReportStep[];
};
```

Правила:

- усі формули рахувати в базових одиницях
- одиниці явно вказувати в `formula` і `items`
- усі величини, які користувач вводить у калькулятор і які мають розмірність, повинні мати змінні одиниці; UI конвертує введене значення в базові одиниці перед викликом ядра, а звіт і результати показують вибрану користувачем одиницю
- `formula` зберігати як plain-text ланцюжок для тестів, accessibility і копіювання
- UI має рендерити цей самий зміст математично, а не як звичайний текст
- нормативне посилання зберігати в `normativeBasis`, окремо від короткого `caption`
- якщо формула є геометричним виведенням із пунктів норм, це прямо вказувати в підписі або специфікації

## Формат кроку звіту

Кожен розрахунковий крок має мати:

```text
<Підпис що обчислюється>:
Нормативна підстава: <пункт і документ>.
<symbol> = <symbolic formula> = <numeric substitution> = <result unit>
```

Приклад:

```text
Визначення другої складової мінімальної площі армування As,min,2:
Нормативна підстава: п. 8.2.1.1 ДСТУ Б В.2.6-156:2010 / п. 9.2.1.1(1), формула 9.1N EN 1992-1-1.
As,min,2 = 0,0013 * bt * d = 0,0013 * 1000 * 450 = 585 мм² = 5,85 см²
```

Для табличних даних:

```text
Визначення середньої міцності бетону на осьовий розтяг fctm за класом бетону:
Нормативна підстава: табл. 3.1 ...
fctm = 2,9 МПа (C30/37)
```

## Мова користувацького звіту

Звіт має читатися як розрахунок, підготовлений інженером, а не як журнал роботи програми.

- Використовуй формулювання `прийнято`, `визначено`, `віднесено`, `перевірено`, `застосування допускається` або `застосування не допускається`.
- Не показуй внутрішні enum-коди, ідентифікатори профілів, назви функцій, «кандидатні позиції», «вихідні рядки», «погоджені матриці» та описи автоматичного режиму.
- Для ручного значення допустимо написати `прийнято користувачем`, якщо це потрібно для аудиту й прямо пояснено, яку нормативну перевірку користувач виконує самостійно.
- Заголовок кроку описує інженерську дію. Нормативне посилання завжди зберігай у `normativeBasis` і показуй окремим рядком `Нормативна підстава: ...`.
- Не дублюй заголовок звіту в екранному або експортованому документі.

## Застосовність перевірок

Перед обчисленням умовного нормативного кроку спочатку перевір його застосовність.

- Якщо умова не застосовується, не показуй формулу як виконану перевірку. Наведи причину й прийняту поправку або наслідок.
- Якщо одна ознака виключає перевірку, це правило має бути однаковим у розрахунковому ядрі, тексті звіту й тестах.
- Знак порівняння у числовій підстановці повинен відповідати фактичним значенням: `≤` для виконаної умови, `>` для невиконаної. Не використовуй сталий знак незалежно від результату.
- Значення, яке не вводиться користувачем, але з'являється у формулі, потрібно або показати серед вихідних даних, або безпосередньо перед формулою пояснити, як його визначено.
- Для кожного позначення напруження, коефіцієнта або проміжного показника дай словесне визначення при першій появі.

## Локалізація й аудиторський слід

- У користувацькому тексті використовуй десяткову кому; canonical plain-text формули та парсери повинні підтримувати її без пошкодження математичного рендерингу.
- Оформлюй складені позначення математичними індексами, а не програмними іменами. Наприклад, `S_tot,base` означає `S` з індексом `tot,base`.
- У тексті використовуй `⇒` або повне речення замість програмного `=>`.
- Узгоджуй слово `бал` з останніми цифрами числа: `бал` для закінчення на 1, крім 11; `бали` для закінчень на 2–4, крім 12–14; `балів` в інших випадках. Наприклад: `1 бал`, `2 бали`, `11 балів`, `21 бал`, `22 бали`, `111 балів`.
- Для табличної перевірки назви всі ознаки, що визначили рядок і графу: групу, матеріал або виріб, товщину, умови експлуатації та нормативний рядок. Результату `дозволено` без цього аудиторського сліду недостатньо.
- Розрізняй табличне нормативне значення та значення, обчислене за формулою. Не замінюй одне іншим округленням без прямої нормативної підстави, зафіксованої в report contract.

## Перевірка математичного рендерингу

Кожну нову літеру, індекс, модуль, оператор і локалізований десятковий запис спочатку додай у тести спільного parser-а. Тест кроку звіту повинен перевіряти точний plain-text рядок, а renderer-тест — відсутність fallback/error markup і коректні математичні індекси. Для формул із модулями та вкладеними індексами виконай також візуальну перевірку у браузері й у DOCX.

## Математичний стиль формул

У UI формули потрібно рендерити математично:

- `As,min` як `A` з індексом `s,min`
- `As,min,1` як `A` з індексом `s,min,1`
- `fctm` як `f` з індексом `ctm`
- `fyk` як `f` з індексом `yk`
- `bt` як `b` з індексом `t`
- `a_s` як `a` з індексом `s`
- `Øs` як `Ø` з індексом `s`

Для формул звіту використовуй `components/calculators/report-formula.tsx`. Компонент парсить контрольований plain-text стиль формули в LaTeX і рендерить підтримані формули через KaTeX. Непідтримані формули повинні безпечно fallback-итись до оригінального plain-text рядка без поломки сторінки.

Plain-text `formula` залишається єдиним canonical formula field у report steps. Не додавай `formulaLatex`, `formulaDisplay` або дубльовані варіанти формули.

Для локальних label, summary і схем використовуй `components/calculators/math-notation.tsx` або локальний renderer, який централізовано перетворює словник позначень у JSX. Не дублюй ручну розмітку одних і тих самих символів у багатьох місцях.

Кожна відрендерена формула має зберігати точний оригінальний `formula` рядок у `aria-label` і `title`, щоб:

- тест перевіряв точний погоджений рядок
- screen reader мав повний текст
- користувач міг скопіювати зміст без втрати символів

Формули у звіті не потрібно обводити рамками або заливками. Основний акцент має бути на підписі кроку та читабельному математичному рядку. Перевірки застосовності норм також використовують спокійний inline-стиль.

## Креслення та позначення

Якщо калькулятор використовує геометричні величини, додай схему-пояснення:

- окремо для кожного типу конструкції або розрахункової схеми
- з підписами тих самих позначень, що й у словнику
- у SVG або іншому контрольованому векторному форматі
- з `role="img"` і зрозумілим `aria-label`

Схема має показувати тільки потрібні для розрахунку величини. Не додавай зайві елементи, якщо вони не використовуються у формулах або вихідних даних.

## DOCX export

Нативні калькулятори, які підтримують експорт звіту, мають мапити наявні `report.steps` у універсальний контракт `DocxReportDocument` з `lib/report-docx/types.ts`.

Canonical formula source залишається plain-text `formula` або `formulas` у кожному кроці звіту. Не додавай дубльовані поля для DOCX-формул у розрахункові модулі. Якщо формула не підтримується DOCX parser-ом, експорт має вставити точний plain-text рядок і не блокувати створення документа.

Для схем передавай контрольований SVG або PNG через `DocxReportFigure`. Browser export конвертує SVG у PNG перед вбудовуванням, тому що static Cloudflare Pages deployment не має server-side document generation.

## Handoff між калькуляторами

Якщо результат одного калькулятора є вихідними даними для іншого, додай пряме посилання з query parameters.

Приклад:

```text
/calculator/rebar-area-bars?minimumArea=678.6&unit=mm2
```

Правила:

- параметри мають бути стабільними й документованими в коді приймаючого калькулятора
- приймаючий калькулятор має безпечно валідувати query parameters
- якщо параметри невалідні, використовуй стандартні значення
- біля посилання потрібно показувати не тільки переданий результат, а й короткий результат приймаючого розрахунку, якщо його можна обчислити тією самою спільною логікою без відкривання іншого калькулятора
- додавай `returnTo` і `returnLabel`, якщо користувач природно має повернутися до попереднього розрахунку
- приймаючий калькулятор має показувати кнопку/посилання назад, якщо `returnTo` валідний
- посилання має бути видимим поруч із результатом, а не тільки внизу сторінки
- тести мають перевіряти і `href`, і фактичне заповнення приймаючого калькулятора

## Валідація

Кожен калькулятор має повертати стабільний звіт навіть при невалідних даних:

- `valid: false`
- список `errors`
- вихідні дані, які користувач ввів
- без `NaN` у формулах

Типові перевірки:

- додатні геометричні величини
- межі застосовності норм
- наявність класів матеріалів у довідниках
- сумісність вхідних параметрів

Якщо один нормативний блок не застосовується, але інший залишається чинним, не зупиняй увесь розрахунок. Додай `warning` і продовжуй ту частину, яка дозволена.

## Тести

Мінімальний набір тестів:

- окремі формули
- табличні значення з довідників
- порядок `steps`
- точні `caption`
- точний `normativeBasis` і його одноразовий рендеринг після `caption` у вебі та DOCX
- точні plain-text `formula`
- граничні умови застосовності норм
- валідація помилкових даних
- UI smoke test через `CalculatorShell`

Тест має спочатку падати, потім проходити після реалізації.

## Реєстрація калькулятора

Для нового native-калькулятора:

1. Додай literal у `nativeCalculator` union в `lib/calculators.ts`.
2. Імпортуй компонент у `components/calculator-shell.tsx`.
3. Додай `case` у `NativeCalculator`.
4. Додай запис у `data/content.json` з `displayMode: "native"`.
5. Перевір `openUrl`, `order`, `useCases`, `tags`, `standard` і за потреби додай `seoContent` для розгорнутих SEO-блоків. `<title>` та `<meta name="description">` генеруються автоматично з `title`, основної категорії та `shortDescription`.
