# Report Formula KaTeX Renderer Design

Date: 2026-06-09
Status: Approved design, pending written spec review

## Goal

Build a shared formula renderer for native calculator reports that keeps one canonical source of truth: the existing plain-text `formula` string generated by each calculation module.

The UI should render supported formulas with KaTeX/LaTeX quality while preserving the original plain-text formula for tests, accessibility, titles, and copy workflows.

## Non-Goals

- Do not add `formulaLatex`, `formulaDisplay`, or other duplicated formula fields to report steps.
- Do not build a full computer algebra system or a parser for arbitrary mathematical writing.
- Do not require all existing formulas to be migrated at once.
- Do not show parser errors to end users.

## Canonical Data Model

Report steps continue to expose a single formula field:

```ts
export type CalculatorReportStep = {
  key: string;
  caption: string;
  formula?: string;
  items?: string[];
};
```

`formula` remains the canonical contract text agreed in report-contract documents and asserted by tests.

## Architecture

Add shared modules:

```text
lib/report-notation.ts
lib/report-formula-parser.ts
components/calculators/report-formula.tsx
```

`lib/report-notation.ts` stores the common notation dictionary that is currently duplicated in calculator components. It covers symbols such as `Mγ`, `Mq`, `Mc`, `φ11`, `φa`, `γc1`, `As,min`, `d1`, and `db,input`.

`lib/report-formula-parser.ts` converts a supported plain-text formula into a small AST or token model and then into LaTeX lines.

`components/calculators/report-formula.tsx` renders the formula:

```tsx
<ReportFormula formula={step.formula} />
```

The component always sets `aria-label={formula}` and `title={formula}`. If parsing succeeds, it renders KaTeX. If parsing fails, it falls back to the existing plain-text style.

## Data Flow

```text
plain-text formula
-> tokenize controlled report syntax
-> parse into formula lines and text fragments
-> convert notation and operators to LaTeX
-> render with katex.renderToString
-> fallback to plain text if unsupported
```

The original formula string is never modified or replaced in the report model.

## Parser Scope

The parser supports the controlled formula style used in report contracts:

```text
lines               := statement ("; " statement)*
statement           := equationStatement | comparisonStatement | assignmentStatement
equationStatement   := expression ("=" expression)+ optionalText
comparisonStatement := expression comparisonOperator expression optionalImplication optionalText
assignmentStatement := symbol "=" expression optionalText
```

Supported syntax:

- numbers: `30`, `30.01`, `1.15`
- symbols with subscripts: `Mγ`, `Mγ,a`, `φ11`, `φa`, `γc1`, `As,min,1`
- operators: `+`, `-`, `*`, `/`, `<=`, `>=`, `<`, `>`, `=>`
- grouping: `(...)`, `[...]`, `min(...)`, `max(...)`
- trailing units: `кПа`, `м`, `т/м²`, `кг/см²`, `мм²`, `см²`
- explanatory text after the mathematical part, for example after `, оскільки` or ` - умова виконується`

Unsupported lines fall back to the plain-text renderer.

## LaTeX Mapping

Notation examples:

```text
Mγ      -> M_{\gamma}
Mγ,a    -> M_{\gamma,a}
Mq      -> M_q
Mc      -> M_c
φ11     -> \varphi_{11}
φa      -> \varphi_a
γc1     -> \gamma_{c1}
As,min  -> A_{s,min}
db,input -> d_{b,input}
```

Operator examples:

```text
*       -> \cdot
<=      -> \le
>=      -> \ge
=>      -> \Rightarrow
(a) / (b) -> \frac{a}{b}
```

For interpolation formulas, a product followed by a parenthesized division renders as a vertical fraction:

```text
(φ11 - φa) / (φb - φa)
```

becomes:

```latex
\frac{\varphi_{11} - \varphi_a}{\varphi_b - \varphi_a}
```

## Expected Interpolation Rendering

The single canonical text:

```text
Mγ = Mγ,a + (Mγ,b - Mγ,a) * (φ11 - φa) / (φb - φa) = 1.15 + (1.24 - 1.15) * (30.01 - 30) / (31 - 30) = 1.15; Mq = Mq,a + (Mq,b - Mq,a) * (φ11 - φa) / (φb - φa) = 5.59 + (5.95 - 5.59) * (30.01 - 30) / (31 - 30) = 5.59; Mc = Mc,a + (Mc,b - Mc,a) * (φ11 - φa) / (φb - φa) = 7.95 + (8.24 - 7.95) * (30.01 - 30) / (31 - 30) = 7.95
```

renders as three KaTeX lines with vertical fractions and mathematical subscripts, while the DOM still exposes the full plain-text string through `aria-label` and `title`.

## Styling

Formula blocks should look like engineering report text, not code blocks:

- no heavy border around each formula
- horizontal scrolling for long formulas
- readable display math size on desktop
- stable line height and no layout shift during render
- graceful wrapping or scrolling on mobile

The component may use shared classes such as:

```text
.report-formula
.report-formula__line
.report-formula__text
.report-formula--fallback
```

Calculator-specific report containers can keep their existing layout while delegating formula display to `ReportFormula`.

## KaTeX Integration

Add the `katex` dependency.

Use `katex.renderToString` from a small React wrapper instead of adding a separate React-specific KaTeX package.

Import KaTeX CSS once at the application level, likely from `app/layout.tsx` or through the global CSS setup.

Render with trusted static strings produced by the parser, not with user-provided raw LaTeX.

## Error Handling

`parseReportFormula(formula)` should return a discriminated result:

```ts
type ParseReportFormulaResult =
  | { ok: true; lines: ParsedFormulaLine[] }
  | { ok: false; reason: string };
```

Production UI ignores `reason` and shows fallback text. Unit tests can assert fallback reasons for unsupported cases.

## Testing

Add parser tests for:

- `Mγ; Mq; Mc` interpolation with vertical fractions
- `R = ... [ ... ] = ... кПа`
- `k = 1.0, оскільки ...`
- `min(...)` and `max(...)`
- comparison and implication operators such as `d1 <= d => ...`
- unsupported input returning a fallback result

Add UI tests for:

- `ReportFormula` preserves `aria-label` as the exact original formula
- supported formulas render KaTeX markup
- unsupported formulas render the fallback text without throwing
- existing native calculator report tests continue to assert exact plain-text formulas

## Migration Plan

1. Add `katex`, `report-notation`, parser tests, and `ReportFormula`.
2. Replace local report formula renderers in native calculators with `ReportFormula`.
3. Start with `soil-design-resistance`, especially table Е.8 interpolation.
4. Gradually remove duplicated regex notation renderers from `cassoon`, `minimum`, and `foundation` calculators where the shared renderer covers the formula style.
5. Update `docs/calculation-reporting-guide.md` to state that formulas are stored only as plain text and rendered through the shared parser plus KaTeX.

## Acceptance Criteria

- Every report step still has at most one canonical formula string.
- No report step stores duplicated LaTeX.
- The table Е.8 interpolation formula renders as multiple KaTeX lines with vertical fractions.
- Exact plain-text formula tests remain meaningful and unchanged in intent.
- Unsupported formulas never break the page.
